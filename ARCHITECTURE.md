# Архитектура сервиса

## Монолит vs микросервисы

Когда я думала над архитектурой, рассматривала два варианта: монолит и микросервисы. Выбрала монолит, потому что для учебного проекта это проще и вполне достаточно.

### Что такое монолит

Всё в одном месте: Flask принимает запрос, берёт модель из памяти и возвращает ответ. Cхема:

```
[Клиент] -> POST /predict -> [Flask + модель] -> JSON-ответ
```

Почему выбрала монолит:
- Просто запустить, просто отлаживать
- Модель уже в памяти при старте, никаких лишних обращений
- Для одной модели усложнять смысла нет

Минусы монолита:
- Если нагрузка вырастет, придётся масштабировать весь сервис целиком
- Обновить модель = перезапустить весь сервис

### Как могло бы быть с микросервисами

Каждая версия модели жила бы в отдельном контейнере, роутер распределял бы трафик между ними:

```
[API Gateway] -> [Model Service v1] \
                                     -> [Router]
              -> [Model Service v2] /
```

Это удобно для A/B-теста, но сразу появляется много сложностей: надо настраивать сеть между сервисами, думать про Kubernetes и так далее. Для данного проекта это избыточно.

Итого: для production с большой нагрузкой микросервисы лучше, но здесь монолит подходит. Две версии модели разделила через параметр `?version=` в запросе, это дало нужную гибкость без лишней сложности.

## uWSGI + NGINX в production

На лекции мы разбирали проблему GIL (Global Interpreter Lock): один Python-процесс может обрабатывать только один запрос одновременно. Для ML-сервиса с большим потоком запросов это серьёзная проблема.

Решение простое: запустить несколько процессов параллельно. Для этого используют связку uWSGI + NGINX:

```
Клиент -> NGINX (порт 80) -> uWSGI (несколько воркеров) -> Flask
```

- **uWSGI** запускает несколько копий приложения. У каждой копии свой интерпретатор Python, поэтому GIL не мешает
- **NGINX** принимает все входящие запросы и раскидывает их по воркерам. Сам Python-код не выполняет, просто очень быстро управляет соединениями

В моём проекте вместо uWSGI использую **gunicorn** - он проще в настройке и работает на всех платформах. В Dockerfile это выглядит так:

```
CMD ["gunicorn", "--workers", "2", "--bind", "0.0.0.0:5000", "app.api:app"]
```

Для полноценного production нужно было бы ещё добавить NGINX как реверс-прокси, это показано в docker-compose.yml.

### ONNX-ML (обзорно)

ONNX - это формат для хранения ML-моделей, который не зависит от фреймворка. Грубо говоря, можно обучить модель в sklearn, сохранить в ONNX, и запускать её уже без Python вообще.

Конвертация нашей модели выглядела бы так:

```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

onnx_model = convert_sklearn(model_v1, "credit_default",
                              [("input", FloatTensorType([None, 23]))])
with open("models/model_v1.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())
```

Зачем это нужно: ONNX быстрее работает на больших батчах данных и позволяет запускать модель в сервисах на других языках, например на C++.

## RabbitMQ (концепт)

Сейчас сервис работает синхронно: запрос пришёл, модель посчитала, ответ ушёл. Это нормально когда запросы приходят по одному.

Но представим: банк хочет каждую ночь прогнать через модель 500 000 клиентов. Делать это через обычные HTTP-запросы плохо: соединение может оборваться, сервер перегрузится.

Тут помог бы RabbitMQ (брокер сообщений):

```
[Банк] -> публикует задание -> [RabbitMQ очередь] -> [ML-воркер] -> результаты в базу
```

Банк кидает задание в очередь и сразу получает подтверждение. Воркер читает задания в своём темпе и обрабатывает батчами. Если что-то упало, RabbitMQ повторит задание автоматически.

## Логирование и мониторинг

Каждый запрос к сервису логируется в JSON:

```json
{"time": "2026-06-12T14:15:04Z", "level": "INFO",
 "msg": {"event": "prediction", "version": "v1", "prediction": 1, "probability": 0.54}}
```

JSON удобен тем, что его легко читать автоматически. В production такие логи собирают через ELK-стек:
- **Logstash** - собирает логи из контейнеров
- **Elasticsearch** - хранит и индексирует
- **Kibana** - рисует дашборды, можно смотреть статистику по предсказаниям в реальном времени

## DVC и MLflow (концепт)

### DVC

Датасет большой, хранить его в Git неудобно. DVC решает эту проблему: сам файл лежит где-то в облаке (S3), а в Git только маленький файл с хэшом.

```bash
dvc init
dvc add data/credit_default.csv
dvc push  # загружаем в облако
```

### MLflow

MLflow помогает отслеживать эксперименты. Когда обучаешь разные версии модели, удобно видеть все метрики в одном месте и сравнивать:

```python
import mlflow
with mlflow.start_run(run_name="logistic_v1"):
    mlflow.log_param("class_weight", "balanced")
    mlflow.log_metric("f1", 0.54)
```

## Бизнес-метрики

Помимо F1 и Precision, банку важны метрики в деньгах:

**1. Сколько денег сэкономила модель**

Если знаем средний убыток от дефолта (например, 50 000 руб.), считаем:

```
Сэкономлено = количество верно найденных дефолтов x 50 000 руб.
```

Чем выше Recall, тем больше дефолтов нашли и тем больше денег сохранили.

**2. Сколько заявок одобряем при допустимом уровне риска**

Можно двигать порог вероятности модели и смотреть, как меняется доля одобренных заявок. Задача банка - одобрять как можно больше заявок, не превышая допустимый уровень потерь.
