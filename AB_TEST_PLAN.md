# План A/B-тестирования

## Зачем вообще нужен A/B-тест

Мы обучили две модели:
- **v1** - LogisticRegression (простая, быстрая)
- **v2** - RandomForestClassifier (сложнее, должна быть точнее)

Чтобы понять, стоит ли переходить на v2, нужно проверить её на реальном трафике. Просто сравнить метрики на тестовой выборке недостаточно - важно убедиться что модель работает хорошо в реальных условиях. Для этого и делают A/B-тест.

---

## Гипотезы

- **H0:** F1-score у v2 такой же как у v1 (разницы нет)
- **H1:** F1-score у v2 выше чем у v1

---

## Как делим трафик

50/50 случайным образом. Каждый новый запрос случайно попадает либо в группу A (модель v1), либо в группу B (модель v2).

В коде это выглядит просто:

```python
import random

def route_to_model():
    return random.choice(["v1", "v2"])
```

В реальной системе такой роутинг делают на уровне API Gateway, чтобы клиент не мог сам выбирать версию. У нас для демонстрации это реализовано через параметр `?version=` в запросе.

---

## Сколько длится тест

Минимум 4 недели. Почему так долго:
- У заёмщиков есть сезонность (зарплаты, праздники и так далее)
- Нужно набрать достаточно событий чтобы результаты были статистически значимыми
- В идеале дольше, потому что дефолт фиксируется через 1-3 месяца после выдачи кредита

---

## Какие метрики смотрим

### Основная: F1-score для класса дефолта

F1 = 2 * (Precision * Recall) / (Precision + Recall)

Выбрала F1 потому что датасет несбалансированный (дефолтов около 22%). Если смотреть только на accuracy, можно просто всегда предсказывать "нет дефолта" и получить 78% - это бесполезная модель.

### Дополнительная: Recall

Recall = сколько дефолтов из всех реальных дефолтов мы нашли

Выбрала именно Recall, а не Precision, потому что пропустить дефолт для банка хуже, чем лишний раз отказать надёжному клиенту. Пропущенный дефолт - это прямые убытки. Лишний отказ - просто упущенная прибыль.

---

## Как считаем результаты

Используем **bootstrap** - это когда многократно (1000 раз) берём случайные подвыборки из наших данных, считаем F1 на каждой, и смотрим на распределение разницы между v1 и v2.

```python
import numpy as np
from sklearn.metrics import f1_score

def bootstrap_f1_diff(y_true_a, y_pred_a, y_true_b, y_pred_b, n_iter=1000):
    diffs = []
    for _ in range(n_iter):
        idx_a = np.random.choice(len(y_true_a), len(y_true_a), replace=True)
        idx_b = np.random.choice(len(y_true_b), len(y_true_b), replace=True)

        f1_a = f1_score(y_true_a[idx_a], y_pred_a[idx_a])
        f1_b = f1_score(y_true_b[idx_b], y_pred_b[idx_b])
        diffs.append(f1_b - f1_a)

    ci_low, ci_high = np.percentile(diffs, [2.5, 97.5])
    return np.mean(diffs), ci_low, ci_high
```

Строим 95% доверительный интервал для разницы F1(v2) - F1(v1). Если весь интервал выше нуля, значит v2 статистически значимо лучше.

---

## Когда переходим на v2

Переходим если выполнены все три условия:
1. F1(v2) выше F1(v1) минимум на 0.02 (2 процентных пункта)
2. Доверительный интервал разницы не включает ноль
3. Recall(v2) не хуже чем Recall(v1)

Если разница меньше 2 процентных пунктов - остаёмся на v1, смысла переходить нет.

---

## Как это влияет на архитектуру сервиса

Чтобы A/B-тест работал, сервис должен:
- Держать обе модели загруженными одновременно (реализовано через MODEL_REGISTRY в api.py)
- Логировать версию модели для каждого запроса (реализовано в api.py)
- Сохранять логи для последующего анализа (JSON в stdout)
