# План A/B-тестирования: сравнение модели v1 и v2

## Постановка задачи

Мы обучили два варианта модели прогнозирования дефолта:
- **v1** - LogisticRegression с `class_weight=balanced` (простая, интерпретируемая)
- **v2** - RandomForestClassifier (сложнее, потенциально выше качество)

Нужно понять, стоит ли переходить на v2 в production. A/B-тест - стандартный
способ проверить это на реальном трафике, не отключая старую модель.

---

## Гипотезы

- **H₀ (нулевая):** F1-score модели v2 не отличается от F1-score модели v1
- **H₁ (альтернативная):** F1-score модели v2 значимо выше F1-score модели v1

---

## Разделение трафика

Случайное распределение **50/50**: каждый входящий запрос случайно назначается
в группу A (v1) или группу B (v2). Это стандартный подход - он обеспечивает
статистическую сопоставимость групп.

На уровне нашего API это реализовано через query-параметр `?version=`:

```python
import random

def route_to_model(request):
    """Простой роутер для A/B-теста."""
    version = random.choice(["v1", "v2"])
    return version
```

В реальной системе роутинг был бы вынесен на уровень API Gateway или nginx,
чтобы клиент не мог сам выбирать версию. Здесь для демонстрации мы оставили
явный параметр.

---

## Продолжительность теста

**Минимум 4 недели.** Обоснование:
- Поведение заёмщиков имеет сезонность (зарплатные дни, праздники)
- Нужно набрать достаточно событий дефолта, чтобы оценить качество
- Как правило, в кредитном скоринге дефолт фиксируется через 1-3 месяца
  после выдачи кредита - поэтому тест должен длиться дольше,
  но для оперативной оценки используют прокси-метрики (задержки платежей)

---

## Метрики оценки

### Основная метрика: F1-score для класса дефолта (класс 1)

F1 = 2 × (Precision × Recall) / (Precision + Recall)

Выбрана как основная, потому что датасет несбалансирован (~22% дефолтов),
и accuracy здесь не информативна - модель может просто всегда предсказывать
"нет дефолта" и получить 78% accuracy.

### Дополнительная метрика: Recall

Recall = TP / (TP + FN)

Я выбрала Recall как дополнительную метрику, потому что в кредитном скоринге
пропустить дефолт (FN) обычно дороже, чем отказать надёжному клиенту (FP).
Пропущенный дефолт = прямые убытки банка. Лишний отказ = потеря прибыли,
но не убыток. Поэтому Recall для нас важнее Precision.

### Бизнес-метрики (описание в ARCHITECTURE.md)
- Ожидаемые предотвращённые потери (ELR)
- Доля одобренных заявок при фиксированном уровне риска

## Статистический анализ

### Метод

Используем **двухвыборочный z-тест для пропорций**, потому что:
- Предсказания модели - это бинарные события (0 или 1)
- Выборки независимы (разные клиенты в группах A и B)
- При достаточном размере выборки (n > 30) z-тест работает корректно

Для F1-score (не пропорция) можно использовать **bootstrap**:
многократно (например, 1000 итераций) случайно выбираем подмножества из
обеих групп, считаем F1 на каждом, строим распределение разницы F1_v2 - F1_v1.

### Пример расчёта (псевдокод)

```python
import numpy as np
from scipy import stats

def bootstrap_f1_diff(y_true_a, y_pred_a, y_true_b, y_pred_b, n_iter=1000):
    """Bootstrap для доверительного интервала разницы F1."""
    diffs = []
    for _ in range(n_iter):
        idx_a = np.random.choice(len(y_true_a), len(y_true_a), replace=True)
        idx_b = np.random.choice(len(y_true_b), len(y_true_b), replace=True)

        f1_a = f1_score(y_true_a[idx_a], y_pred_a[idx_a])
        f1_b = f1_score(y_true_b[idx_b], y_pred_b[idx_b])
        diffs.append(f1_b - f1_a)

    ci_low, ci_high = np.percentile(diffs, [2.5, 97.5])
    return np.mean(diffs), ci_low, ci_high

mean_diff, ci_low, ci_high = bootstrap_f1_diff(...)
print(f"Разница F1: {mean_diff:.4f}, 95% CI: [{ci_low:.4f}, {ci_high:.4f}]")
```

### Доверительные интервалы

Используем 95% доверительный интервал (α = 0.05). Если CI для разницы
F1_v2 - F1_v1 полностью выше нуля - v2 статистически значимо лучше.

### Критерий успешности

Переходим на v2, если выполнены **оба** условия:
1. F1(v2) > F1(v1) + **0.02** (улучшение не менее 2 процентных пунктов,
   чтобы разница была практически значимой, а не только статистически)
2. 95% доверительный интервал разницы не включает 0
3. Recall(v2) ≥ Recall(v1) (не ухудшаем главную бизнес-метрику)

Если v2 хуже или не даёт значимого улучшения - остаёмся на v1.

## Влияние A/B-теста на архитектуру сервиса

A/B-тест накладывает требования на сервис:
- Обе модели должны быть загружены одновременно (реализовано: MODEL_REGISTRY)
- Каждый запрос должен логировать версию модели (реализовано в api.py)
- Нужно хранить лог запросов для последующего анализа (JSON-логи в stdout)

Если бы тест проходил долго или моделей было много, имело бы смысл вынести
роутинг в отдельный микросервис - это одна из причин, почему при масштабировании
монолит уступает микросервисам.
