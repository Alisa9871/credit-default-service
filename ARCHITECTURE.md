# Архитектура сервиса

## Монолит vs микросервисы

При разработке этого проекта я рассматривала два варианта архитектуры:
монолитный и микросервисный. В итоге был выбран **монолитный подход**, и вот почему.

### Монолитная архитектура (выбранный вариант)

Всё приложение - один процесс: Flask читает запрос, загружает модель из памяти
и возвращает ответ. 

**Плюсы для данного проекта:**
- Просто запустить и отлаживать 
- Модели уже загружены в памяти при старте, нет задержек на обращение к другому сервису
- Не нужно настраивать сеть между сервисами
- Для учебного проекта с одной моделью - это оправданно

**Минусы:**
- Если нагрузка возрастёт, масштабировать можно только сервис целиком
- Модель и API жёстко связаны - обновить модель = перезапустить весь сервис

### Микросервисная архитектура (альтернатива)

Могла бы выглядеть так:

Каждая версия модели - отдельный контейнер. Роутер распределяет трафик между
ними (например, 50/50 для A/B-теста). Но это сразу добавляет:
- Оркестрацию (Kubernetes или как минимум Docker Compose с несколькими сервисами)
- Сетевые задержки между сервисами
- Сложность деплоя и мониторинга

**Вывод:** для production с высокой нагрузкой и несколькими моделями микросервисы
лучше. Но для данного учебного проекта я считаю, что монолит более правильный выбор. Два режима
модели (v1/v2) разделены через query-параметр `?version=`, что даёт гибкость
без лишней сложности.

## uWSGI + NGINX в production

На лекции по теме GIL мы разобрали ключевую проблему Python-серверов: из-за
**Global Interpreter Lock (GIL)** один процесс Python может обрабатывать только
один запрос одновременно. Это критично для ML-сервиса, который получает много
параллельных запросов.

**Решение — запустить несколько процессов:**

- **uWSGI** запускает несколько копий приложения (воркеры), каждая копия
  независимо обрабатывает запросы. Ограничение GIL обходится, потому что
  у каждого воркера свой интерпретатор Python.

- **NGINX** принимает входящие HTTP-запросы и передаёт их в uWSGI через
  бинарный протокол `uwsgi` (через UNIX-сокет). NGINX при этом не выполняет
  никакого Python-кода — он просто очень эффективно управляет соединениями
  (асинхронная архитектура, один процесс — тысячи соединений).

В нашем проекте вместо uWSGI используется **gunicorn** (принцип тот же, но
проще в настройке и доступен на всех платформах). В Dockerfile:

Для полноценного production понадобилось бы добавить NGINX как реверс-прокси
(как показано в docker-compose.yml).

### ONNX-ML (обзорно)

ONNX (Open Neural Network Exchange) — это стандартный формат для хранения
ML-моделей, независимый от фреймворка. Конвертация нашей sklearn-модели
выглядела бы так:

```python
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType

onnx_model = convert_sklearn(model_v1, "credit_default",
                              [("input", FloatTensorType([None, 23]))])
with open("models/model_v1.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())
```

**Зачем это нужно:** ONNX Runtime умеет выполнять предсказания быстрее, чем
стандартный sklearn predict, особенно на больших батчах. Также ONNX позволяет
запускать модель на устройствах без Python (например, в C++ сервисе).

## Брокер сообщений (концепт RabbitMQ)

Текущий сервис синхронный: клиент отправил запрос → ждёт ответа → получает.
Это нормально для единичных предсказаний в реальном времени.

Но представим сценарий: банк хочет каждую ночь прогонять через модель 500 000
клиентов (батч-скоринг). Делать это синхронно через HTTP - плохая идея:
соединение может оборваться, нельзя отследить прогресс, сервер перегружается.

**Решение с RabbitMQ:**

```
[Банковская система]
        ↓  publish(client_ids)
[RabbitMQ очередь]
        ↓  consume()
[ML Worker — наш сервис]
        ↓  результаты
[Database / другая очередь]
```

Банк публикует задание в очередь и сразу получает подтверждение. ML-воркер
читает задания в своём темпе, обрабатывает батчи по 1000 клиентов, записывает
результаты в базу. Если воркер упал — RabbitMQ повторит задание автоматически.

Такой подход также подходит для **логирования**: каждый predict-вызов публикует
событие в очередь, отдельный consumer сохраняет их в ELK-стек (Elasticsearch +
Logstash + Kibana) для мониторинга и аудита.

## Логирование и мониторинг

Сервис логирует каждый запрос в JSON-формате:

```json
{"time": "2026-06-12T14:15:04Z", "level": "INFO",
 "msg": {"event": "prediction", "version": "v1",
         "prediction": 1, "probability": 0.54}}
```

JSON-формат выбран, потому что его легко парсить автоматически. В production
эти логи можно передавать в **ELK-стек**:
- **Logstash** - собирает логи из файлов или stdout контейнера
- **Elasticsearch** - индексирует и хранит их
- **Kibana** - визуализирует: дашборд с долей дефолтов по времени, аномалии
  в вероятностях, латентность эндпоинта

## MLOps-инструменты (концепт)

### DVC (Data Version Control)

Проблема: датасет большой и хранить его в Git - плохая практика. DVC решает это:
он отслеживает изменения в данных и моделях, храня сами файлы в S3/GCS, а
в Git-репозитории — только маленькие `.dvc`-файлы с хэшами.

Для нашего проекта это выглядело бы так:

```bash
dvc init
dvc add data/credit_default.csv   # файл уходит в .gitignore, dvc отслеживает хэш
dvc push                           # загружаем в S3
```

### MLflow

MLflow - инструмент для отслеживания экспериментов. При обучении мы логируем
параметры и метрики:

```python
import mlflow
with mlflow.start_run(run_name="logistic_v1"):
    mlflow.log_param("class_weight", "balanced")
    mlflow.log_metric("f1", 0.54)
    mlflow.sklearn.log_model(model_v1, "model")
```

Потом в UI MLflow видно все эксперименты, можно сравнить v1 и v2 по F1,
Precision, Recall и выбрать лучшую для деплоя.

## Бизнес-метрики

Помимо F1 и Precision, для банка важны бизнес-метрики:

**1. Ожидаемые предотвращённые потери (Expected Loss Reduction)**

Если банк знает средний убыток от одного дефолта (например, 50 000 руб.),
то можно посчитать, сколько денег экономит модель:

```
ELR = TP × avg_loss_per_default
```

где TP - количество правильно предсказанных дефолтов. Чем выше Recall модели,
тем больше дефолтов поймали и тем больше денег сохранили.

**2. Доля одобренных заявок при том же уровне риска**

Можно варьировать порог вероятности (threshold) модели и смотреть, сколько
заявок одобряем при заданном уровне ожидаемых потерь. Модель v2 с лучшим
Precision позволяет одобрить больше заявок при том же числе ожидаемых дефолтов.

```
Approval Rate = (TN + FN) / total_applications
```

Задача банка - максимизировать Approval Rate при ограничении на допустимые потери.
